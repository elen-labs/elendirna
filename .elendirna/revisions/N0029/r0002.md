## Delta

## FAQ: "그냥 Git 쓰면 안 되나요?"

설계가 진행될수록 Git과 유사한 구조가 나타나고 있다 — revision/baseline, index, sync log, author, remote transport.
이는 "변경 이력이 있는 분산 저장소" 문제의 자연스러운 수렴이다.

### 유사한 것들

| Elendirna | Git |
|-----------|-----|
| revision + baseline chain | commit + parent |
| index.sqlite | .git/index |
| sync.jsonl | reflog |
| bundle --since | git log --since |
| elf validate | git fsck |
| author 필드 | commit author |
| multi-vault | remote repository |
| SSE Transport | git daemon / smart HTTP |

### 그럼에도 Git이 아닌 이유

**버전 단위가 다르다.**
Git은 파일을 버전 관리한다. Elendirna는 *아이디어*를 first-class object로 관리한다.
파일 구조 변경 없이 생각이 바뀌면 — Git은 기록하지 못한다.

**delta의 의미론이 다르다.**
Git diff는 구조적(줄 단위). Elendirna의 delta는 의미적 — "왜 이 생각이 바뀌었는가"를 담는다.
이건 사람이 작성하는 서술이지, 자동 생성되는 차이가 아니다.

**AI-native 레이어가 없다.**
sync_record, bundle, MCP 통합 — 에이전트가 vault를 능동적으로 탐색하고 세션 간 컨텍스트를 복원하는 워크플로는 Git에 없다.

**요약:**
Git이 검증한 구조를 지식 관리 + AI 워크플로에 맞게 재해석하는 방향이다.
수렴은 올바른 설계의 증거이지, 차별성의 부재가 아니다.
